Experimental results¶

The training configurations are the same as previous report. Here precision@k are provided. Please note that the reported scores may be a little bit different from the previous ones (but not significant), because of random initialized parameters (the minor difference still exits even though the random seed is fixed). The precision@k and recall@k are calculated using sklean.metrics for fair comparison. To compute precision and recall, a hyper-parameter average needs specification. Two choices, i.e., micro and marco are provided. By selecting the global parameter average, the corresponding values will be displayed.

def simple_table(scores, max_len, average="micro", metrics="precision"):
    
    key = "%s_%s" % (metrics, average)
    item_list = list(scores.keys())
    header = ["%s@%s" % (metrics[0].upper(), i) for i in range(1, max_len+1)]
    
    data = []
    for item_scores in scores.values():
        value = []
        for score in item_scores.values():
            value.append(score[key])
        if len(value) < max_len:
            value += ["-"] * (max_len-len(value))
        data.append(value)
    
    # print row format
    row_format ="{:>10}" * (len(header) + 1)
    print(row_format.format("", *header))
    for item, row in zip(item_list, data):
        print(row_format.format(item, *row))

Experimental results for testing impact of text length on performance¶

It is mentioned in the Summary of data that the maximum length is 35, with median value 13. In fact, all long texts are tokenized into short ones based on fullstops. The reason is that CNN models require a maximum length, and paddings will be added if the feeding text is shorten than the maximum. Allowing longer text on one hand increases model complexity (thus a wider network), on the other hand, paddings will become dominant for very short text. In this regard, we do not really have long text. Anyway, we could still divide texts into two groups based on their legnths. In this experiment, we choose 13 as the cut-off length. The bar plot is shown as below. It is clearly that the recall for predicting long texts is generally higher than those for short texts. It could be two potential reasons:

  1. Most of training examples are categoried as long texts (longer than 13);
  2. Long texts contain explict information for the classification.

However, if the text is long enough, e.g., two problems would arise: 1) it is out of the model capacity; and 2) the predicted performance will be poor.

def long_short_text_plot(scores, average="micro"):
    items = []
    short_scores = []
    long_scores = []
    for item, value in scores.items():
        items.append(item.upper())
        short_score = value["short"]["recall_%s" % average]
        long_score = value["long"]["recall_%s" % average]
        short_scores.append(float(short_score))
        long_scores.append(float(long_score))
    
    print("recall for short texts:", short_scores)
    print("recall for long texts:", long_scores)
    
    # create plot
    n_groups = 5
    fig, ax = plt.subplots()
    index = np.arange(n_groups)
    bar_width = 0.35
    opacity = 0.8

    rects1 = plt.bar(index, short_scores, bar_width,
    alpha=opacity,
    color='b',
    label='Short')

    rects2 = plt.bar(index + bar_width, long_scores, bar_width,
    alpha=opacity,
    color='g',
    label='Long')

    plt.xlabel('Item')
    plt.ylabel('Recall')
    plt.xticks(index + bar_width, items)
    plt.legend()

    plt.show()

Expriment on data imbalance¶

According to data distribution, the data is highly imbalanced. In roder to test CNN's sensitivity on data imbalance, we re-group the training data. Here is strategy:

  1. divide the total dataset into 2 or 3 groups, based on the number of certain labeled texts. In each group, the labeled text for each class is more balanced than without grouping.
  2. Train & test model for each group. The data is divided into train/test set based on 0.9/0.1 ratio. No dev data is generated. All model & training configurations are the same except selecting training epochs 3.

We choose CPU for example. Here are the data distribution: { "6": 187, "2": 357, "0": 194, "1": 130, "7": 497, "5": 54, "8": 206, "9": 39, "4": 55, "3": 71 }. Then labels (7, 2, 8, 0, 6) and (1, 3, 4, 5, 9) are grouped together to generate two seperated datasets, and labels (7, 2, 8), (0, 6, 1) and (3, 4, 5, 9) are grouped to generate three seperated datasets.

One problem is that the training data will be fewer after re-grouping. Intuitively, groups where training data are sufficient will obtain better results than the groups with insufficient data, even though the data are distributed evenly accross labels. Results are shown below.

def group_plot(item="cpu", k=None, metrics="recall", average="micro", group_index=0, num_class=10):
    assert item == "cpu"
    ks = [str(i) for i in range(1, num_class+1)]
    if k is not None:
        ks = k.split(",")
    
    scores_1 = load_json("output/scores.json")
    scores_23 = load_json("output/by_group/scores.json")
    
    legend_text = []
    for k in ks:
        try:
            k_score_1 = float(scores_1[item][k]["%s_%s" % (metrics, average)])
            k_score_2 = float(scores_23["%s_2%s" % (item, group_index)][k]["%s_%s" % (metrics, average)])
            k_score_3 = float(scores_23["%s_3%s" % (item, group_index)][k]["%s_%s" % (metrics, average)])
            k_scores = [k_score_1, k_score_2, k_score_3]
            legend_text.append(metrics[0].upper()+"@"+k)
            index = [1, 2, 3]
            plt.plot(index, k_scores, "-o")
        except:
            pass
    plt.legend(legend_text)
    plt.xticks([1, 2, 3], [1, 2, 3])
    plt.xlabel("Group by label frequency")
    plt.title("%s/2 Group" % (group_index+1))
    plt.show()
